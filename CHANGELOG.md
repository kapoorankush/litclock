### Changed

A fielded/gift LitClock must never apt-upgrade behind the owner's back — a surprise kernel or driver bump could break the e-ink stack with nobody at the keyboard. Sample weather coordinates in `README.md` / `env.sh.sample` are rounded to city-block precision (`30.27, -97.74`), and the full-precision latitude that appeared in redaction test fixtures, a code comment, and an earlier CHANGELOG entry is replaced with an arbitrary example value — the repo's own diagnostics-redaction standard (2-decimal rounding) now applies to the repo itself. - **Example city decoupled from the maintainer's location** (#82 PII sweep, part 2). ## [v0.218.0] - 2026-07-11

### Changed
- **BREAKING: the Control PWA is now at `http://litclock.local` / `http://<ip>` — no port to type** (#343). control_server moved from port 8443 to **port 80**, so the URL a recipient scans from the e-ink QR, taps from the mDNS bookmark, or types by hand no longer carries a `:port` — a smaller, less technical-looking address for non-technical gift users. (This supersedes #343's original 8080 proposal: 8080 would still have shown a port; only 80 removes it.) The port is bound by the non-root `pi` service account via a one-line sysctl drop-in (`/etc/sysctl.d/30-litclock-unprivileged-ports.conf` → `net.ipv4.ip_unprivileged_port_start=80`), deliberately chosen over `AmbientCapabilities=CAP_NET_BIND_SERVICE`: the `litclock-control` unit runs `NoNewPrivileges=no` + setuid `sudo` for the reboot/poweroff actions, and adding a capability directive to a `User=pi` unit has repeatedly flipped the kernel `NoNewPrivs` bit and broken setuid sudo on this hardware — a sysctl never touches that wiring (and stays safe once #82 drops the `010` blanket sudo, when setuid `020` sudo becomes the only reboot path). `setup_server.py` stays HTTPS on 8443 (the iOS captive trust dance needs TLS) — the two are now different ports for different phases, no longer sharing 8443. The port + URL are a single source of truth (`src/control_url.py::control_base_url`, which omits `:80`), shared by control_server (bind) and the e-ink clock (QR); the mDNS probe in `status.js` derives the port from the live origin — so the QR target, the bookmark switch, and the actual listen port can never drift. Installed on fresh flash (pi-gen), manual install (`install.sh`), and OTA (`update.sh` applies the sysctl live before restarting control_server).
  - **Migration (one-time, recoverable):** on OTA update the e-ink QR refreshes to the port-less URL within one paint cycle. A PWA already pinned to a phone home screen, or a saved `:8443` bookmark, will break (the origin includes the port) — **re-scan the QR (it's right there on the clock) and re-pin / re-bookmark.** No data is lost.

## [v0.217.0] - 2026-07-10

### Added
- **Diagnostics: a "Download full logs" export for shell-less support** (#416 follow-up). The Control PWA's diagnostics "Copy support payload" already hands a helper the system state + a **3-line** journal preview per unit — deliberately shallow so it never blocks first paint. That's often too thin to actually debug a failing unit. New `GET /api/diagnostics/support-logs` assembles a single downloadable `text/plain` bundle: the same redacted system payload plus a **~50-line** journal tail per unit across the `DIAG_UNITS` allowlist, so a non-technical owner can hand over one file (or paste) that carries enough context — no SSH. Off the page/poll critical path; deep reads use a **distinct, line-count-scoped cache key** so they never serve or poison the 3-line page-preview cache; a wall-clock budget bounds the serial `journalctl` loop and appends an explicit *named-units* truncation note rather than silently dropping units; `no-store` + redacted (secrets in journal lines are stripped). The existing per-unit `GET /api/diagnostics/journal` also gained a capped `?lines=N` param (default 3, max 200) for pulling a deeper tail for a single unit. Surfaced as a plain `<a download>` link beside the Copy button, so it works with no JavaScript.

- **LKG auto-revert: the clock now self-heals a bad over-the-air update without SSH or an SD reflash** (#209 follow-up). #209 shipped the *writer* (`litclock-lkg.service` records the last SHA that actually painted to `/var/lib/litclock/lkg-sha`, heartbeat-gated); this ships the *consumer*, `litclock-bootcheck.service`. It answers one question per boot — "did the clock paint a frame since this boot?" — via the tmpfs render heartbeat (network-independent, so a merely-offline clock is never touched). On a persistent failure it self-heals: fail 1 and 2 auto-reboot to retry (a single failed render may be transient), and on fail 3 it pins the last-known-good SHA and routes recovery **back through `update.sh`** in a new rollback mode, which does the *complete* install (git + submodules + venv hash-gate + units→`/etc` + sudoers + dispatcher + smoke) rather than a partial `git reset` — so a dependency- or unit-level bad update is actually recoverable, not just a code-only one. If the recovered code also fails it gives up (a persistent `bootcheck-gave-up` marker + best-effort "please re-flash" splash) instead of looping, bounding the whole sequence to ~4 reboots. Two supporting fixes make the recovery target reliable: `update.sh` **no longer clears `lkg-sha`** at the start of an update (the heartbeat-gated writer replaces it only once new code paints, so `lkg-sha` always points at the last code that actually rendered — a dead-on-arrival update can never blank the recovery target, closing the DOA gap), and a `blocked-sha` suppression stops the weekly timer from re-installing the exact release bootcheck just reverted from until a newer release supersedes it. Runs as `pi` using only the `systemctl reboot` + `systemctl start --no-block litclock-update.service` grants already in `sudoers/020_litclock-control` — no new privilege, works after #387 drops `010`. Hard prerequisite for #82 (public release: SSH is off, so a recipient's bricked clock would otherwise need a physical reflash). Design reshaped in `/plan-eng-review` + Codex outside-voice (plan: `docs/plans/lkg-bootcheck-plan.md`).

### Security
- **Closed the pi→root escalation paths so dropping the blanket `010_pi-nopasswd` sudo is safe** (#387, prerequisite for #82). Each was harmless while `010` grants `pi` NOPASSWD:ALL, but becomes a live hole the moment the public image ships with only the scoped `020_litclock-control` allowlist. **(tz-wrapper)** `geocoding.set_system_timezone` (the IP-geo + A18 browser-tz fallback) ran `sudo timedatectl set-timezone <arbitrary-tz>`, which `020` could only authorize via a `set-timezone *` glob — a hole letting `pi` set the clock to any string reaching the resolver. It now calls a new root-owned wrapper `/usr/local/lib/litclock/litclock-set-timezone` that re-validates the timezone against the kernel zoneinfo list *in root-owned code* (a metachar pre-filter + `grep -Fxq`, and `timedatectl` resolved via PATH — never an env-var binary override that could cross `sudo`); `020` authorizes just that fixed path. **(C1)** the root NetworkManager dispatcher invoked the **pi-writable** `/home/pi/litclock/scripts/litclock-mark-collected.sh` — pi→root on the next DHCP event once `010` drops. It now runs a root-owned copy in `/usr/local/lib/litclock/` (no fallback to the repo copy). **(prepare-for-gift)** `litclock-prepare-for-gift.service` (root, startable by `pi` via `020`) exec'd the **pi-writable** `reset-setup.sh` — pi could edit it and get arbitrary root code. It now execs a root-owned copy (with its sourced `lib/state.sh` shipped root-owned alongside), and the gift-message step runs the root-owned system `/usr/bin/python3` instead of the pi-writable venv interpreter. **(NTP grant)** `first-boot.sh`/`install.sh` run `sudo timedatectl set-ntp true`, which `020` didn't authorize — a fresh clock would silently never sync time once `010` drops; added as a fixed-argv grant. **(C2)** the dispatcher + mark-collected writer run as root but write into the `0755 pi pi` `/run/litclock` and `/var/lib/litclock` dirs; both now refuse to follow a symlink at the target and stage via `mktemp` (O_EXCL) instead of a guessable `$MARKER.tmp.$$` path. (A residual sub-millisecond check-then-open race remains — closing it fully needs an `O_NOFOLLOW` open POSIX `sh` can't express — accepted as an explicit risk: it requires an already-compromised `pi` shell winning a race for a limited-content root write.) All root-owned helpers install root:root across pi-gen, `install.sh`, and `update.sh`; the gift-mode `timedatectl set-timezone UTC` fixed-argv path is unchanged.

- **Diagnostics redaction now rounds compound-key and odd-shaped coordinates in the support-logs / journal export** (#497, #498). The diagnostics redaction rounds coordinates to ~city-block precision so the copy / support-logs / journal surfaces stay safe to paste into a public GitHub issue — but `_COORD_KEYED_RE` led with `\b`, which cannot match inside a compound key (the `_` before `LATITUDE` is a word char, so there is no boundary). systemd logs `env.sh` verbatim when a line uses `export` (`Ignoring invalid environment assignment 'export WEATHER_LATITUDE=33.1234'`), so the device's **full-precision, street-level home coordinates** rode un-rounded into the exported bundle while the structured field correctly showed the rounded `33.12`. The lead-in is now a zero-width negative lookbehind (`(?<![A-Za-z])`) — matching how the sibling `_PSK_RE` handles compound keys — which fixes the compound key, *also* catches back-to-back coordinates with no separator (a consuming lead-in would eat the boundary char, leaving nothing for the next coord's lead-in), and is safer against false positives (a *letter* before the keyword still blocks `belong=` / `along=` / `flat=` / `translation=`). The value group additionally rounds signed and scientific-notation spellings and JSON quoted-key forms (`{"lat": …}`); comma-decimals are deliberately left un-matched, since the `_validate_latitude` writer's `float()` rejects them (so they are unreachable) and a comma is ambiguous with a list separator (`lat=1,2,3` must never be fabricated into a coordinate). Found via a coordination-protocol pentest of the Control PWA (blind executor + blind validator, then a Codex adversarial `/review`); every non-plain-decimal shape was verified non-reachable in-tree, so the value-shape work is defense-in-depth. No incremental LAN exposure — the PWA is unauthenticated on the LAN by design and already serves coordinates rounded by default; the real fix is the share-safety contract (exported support-logs are meant to be safe to post publicly).

## [v0.216.0] - 2026-07-07

### Added
- **Printable 4-page gift booklet** (#461, #404). A foldable "little book about your clock" to enclose when LitClock is given as a gift — read in five minutes, no jargon, black-and-white-printer-friendly. Source of truth is `docs/manual/litclock-manual.html`; `docs/manual/build.sh` renders two booklet PDFs (A4 + US Letter) that each fold from one double-sided sheet. Visual register matches the Control PWA / `DESIGN.md` (Fraunces + Instrument Sans, warm paper). A build-time page-count guard fails loudly rather than committing a wrong-sized or overflowed PDF.

### Changed
- **The "Switch to litclock.local" bookmark prompt now verifies it's really talking to your clock** (#487, #309). The mDNS probe that decides whether to offer the address-stable `litclock.local` URL used an opaque `no-cors` fetch of `litclock.local:8443/api/health` — it could only tell that *something* answered on that port, not that *your LitClock* answered, so a different `.local` device on the LAN could false-positive the offer and strand you on a stranger's page. `/api/health` now returns an `app: "litclock"` identity marker with an `Access-Control-Allow-Origin: *` header (read-only, no secrets, no credentials, scoped to that one endpoint), and the probe does a readable fetch that reveals the card only when the response confirms a real LitClock. It fails closed on any error, non-JSON, or mismatched identity. Cross-model `/review` (Codex + Claude) confirmed the wildcard CORS adds no exposure beyond the already-unauthenticated LAN surface and the probe stays a preflight-free simple GET.
- **`/diagnostics` journal tails now hydrate per-unit, off the render path** (#436, #475). The page used to block first paint on a synchronous `journalctl` tail for every unhealthy systemd unit — a cold journal read measured **5–7 s** on a Pi Zero 2W, so `/diagnostics` hung by that much exactly when a unit had failed (i.e. when you opened it to debug), and erratically, since the persistent-journal page cache evicts under memory pressure. Now `_build_service_states` fetches no tails; each non-healthy row hydrates its own tail from a new `GET /api/diagnostics/journal?unit=<unit>` endpoint (validated against the `DIAG_UNITS` allowlist, redacted, `no-store`) **after** first paint, so the page is interactive immediately and one slow unit can't stall another's tail (the multi-failure case that previously blew the client's 10 s budget). The tail cache TTL is decoupled to 45 s (> the 30 s poll) so a stuck-failed unit doesn't re-fork `journalctl` every poll; the copy support payload appends the client-hydrated logs **inside** its fenced block; a no-JS browser sees status chips + a `<noscript>` hint. Cross-model `/review` (Codex + Claude specialists) caught and fixed a stale-response race (per-unit generation guard + a recovered-row guard), a per-poll "loading" flash, and the copy-fence placement.
- **Static PWA assets now send a 15-minute `Cache-Control` max-age** (#465, #436). control_server's static handler shipped an ETag but no `max-age`, so every PWA navigation re-validated ~9 CSS + ~9 JS + 2 web-fonts against the device — a per-navigation 304-revalidation storm, worst on iOS Safari at the plain-HTTP private-IP origin where the service worker never registers, so the HTTP cache is the only layer. A 15-minute `max-age` (deliberately short, since static URLs aren't fingerprinted and a long cache would serve stale CSS after an in-place `update.sh`) removes the repeat-navigation cost.
- **Healthy oneshot service rows at rest now read green, not neutral** (#464, #463). A `DIAG_ONESHOT_UNITS` member settled at its by-design `inactive` resting state now renders the faint-green `settled-ok` chip tone (matching the OK section pill) instead of neutral graphite, so the Services drawer stops implying "attention needed" for a perfectly healthy at-rest oneshot. `/review` caught and fixed a dark-mode-parity miss (via an enum-completeness scan) and folded in an unrelated date-bomb fix (a hardcoded test timestamp that had aged past a 7-day staleness threshold).
- **Manual: a "Where to keep it" placement note on the Getting-started page** (#476). Sets the read-up-close expectation — LitClock is meant to be read like a book, not glanced at from across the room like a wall clock — so a first-time recipient doesn't expect to read the time from a distance.
- **Update-check GH-API cache moved off the SD card onto tmpfs** (#434). `update-check.json` (the 6h-TTL cache of the latest release tag + CHANGELOG that `/api/update/check` and `update.sh` share) used to live in the persistent `/var/lib/litclock/` state dir. It's a purely derived blob, so it now lives on the `/run/litclock` tmpfs alongside the weather cache — a reboot clears it and the next check simply refetches once. The must-persist state (`lkg-sha`, `update-failed`, `post-update-grace-until`, the `last-update` mirror) stays in `/var/lib/litclock`. `update_state.py` (`DEFAULT_CACHE_FILE`) and `update.sh`'s Phase 7 invalidation `rm` point at the new path in lockstep, and both honor the **same** `LITCLOCK_UPDATE_CHECK_CACHE` full-path override so a test harness that repoints one consumer repoints both (a lockstep regression test now guards the default path). `update.sh` also sweeps the pre-#434 SD-resident copy on the next update, so an in-place upgrade doesn't leave a stale blob on the flash card. The writer was already best-effort + atomic (graceful `parent.mkdir`, mv-tmp `os.replace`). Second of the #434 SD-wear reductions.
- **Weather response cache moved off the SD card onto tmpfs** (#434). The provider cache (`weather-cache-{provider}-{units}-{lat}-{lon}.json`) used to write to the SD-backed project root (`/home/pi/litclock/`) once per `WEATHER_TTL` (default 1h) — ~8,760 flash writes/yr when weather is enabled, the largest recurring app-layer SD writer. It's a purely derived blob (re-fetched on demand), so it now lives in the pi-owned `/run/litclock` tmpfs dir created at boot by `systemd/tmpfiles.d/litclock.conf` (the same dir as the render heartbeat + status files); a reboot clears it, which just triggers one fetch on the next tick. The unit/lat/lon path-traversal validation is unchanged (defense-in-depth before the filesystem write). The write is now **best-effort + atomic**: persisting happens after the fetch succeeds and via `tempfile`+`os.replace` (matching `_write_status_file`), so a cache-write failure logs and returns the live data instead of sinking the render, and a crash mid-write can't leave a truncated JSON. The orphan sweep scans the legacy SD project root **once per process** (a one-time migration that drops any stale flash-resident cache on the first fetch), then only the tmpfs dir on subsequent ticks — so upgraders are cleaned up without readdir-ing the SD card every hour. `reset-setup.sh` now also clears the tmpfs copy. Overridable via `LITCLOCK_WEATHER_CACHE_DIR` for dev/non-Pi hosts. First of the #434 SD-wear reductions (weather cache is the app-layer item; journald/OS-level items still pending a decision + hardware QA).

### Fixed
- **Fresh-flash clock never left the "Ready to read." handoff splash** (#388, #484). On a freshly imaged Pi the clock got stuck on the handoff splash forever: handoff itself completed fine (marker written, location + timezone resolved), but `litclock.service` (the per-minute quote painter) died every minute with lgpio `'GPIO busy'` so no quote ever painted. Root cause: `control_server` (`litclock-control.service`) is **long-lived** and painted the handoff splash **in its own process** — and the lgpio line claims a paint makes are held for the whole process lifetime (`epd.sleep()` only deep-sleeps the panel; `module_exit()`/gpiozero-close does not free the claims — only process exit does). So the always-on PWA held the e-ink GPIO forever and the quote painter could never claim it. (authorclock never hit this because its control_server had been restarted post-setup, freeing the GPIO; only a pristine fresh flash exposes it.) The splash now paints via a **short-lived subprocess** (the same pattern `first-boot.sh` already uses for the hotspot/status splashes) that claims, paints, and frees the lines on exit — `render_eink_splash` no longer touches the hardware in-process. A cross-model `/review` hardened two follow-ups: the `handoff-splash` CLI now exits non-zero on a failed paint (so a real display fault surfaces instead of a silent success), and the timeout handler no longer logs the subprocess argv (which carried the SSID + location name) into the diagnostics log.
- **On-screen setup copy no longer names "Safari"** (#482, #485). The e-ink hotspot/setup splash (`"Or open Safari:"`) and the iOS captive-portal bridge page (`"Open in Safari"`) hard-coded the iOS browser name, which is wrong on Android / any non-iOS phone. Both now say "browser" generically; the bridge instruction was rephrased to not quote a specific button label at all (iOS's actual captive-assistant escape button is labelled "Open in Safari", so a quoted "Open in Browser" would misdescribe the only control that page is shown to). Found in fresh-flash test-Pi QA.
- **iOS captive portal now auto-opens on a fresh flash** (#483, #489). Fresh-flash test-Pi QA (with the #486 `CAPTIVE-PROBE` diagnostic in the image) pinned the real cause from the hotspot's dnsmasq log: iOS 17+ sends an HTTPS-RR (type 65) query for `captive.apple.com` before the A query (RFC 9460), and NetworkManager's shared-mode dnsmasq — which reads `/etc/resolv.conf` and inherits a public upstream like `8.8.8.8` — **forwarded** that type-65 query upstream. The isolated hotspot has no route to that upstream, so the forward failed and dnsmasq returned `REFUSED (EDE: network error)`, which iOS reads as hostile DNS and silently demotes the captive sheet (the exact failure #178 was chasing). The `local=/#/` mitigation never actually stopped the forward — it only looked fine on client Pis whose inherited upstream happened to be reachable and answered NODATA. The fix adds **`no-resolv`** to the captive dnsmasq config so it keeps no upstream at all and answers every non-A type authoritatively as NODATA; iOS falls through to the A record and pops the sheet. Manual "open in your browser" always worked (it only needs the A record), which had masked the DNS cause. Reproduced + fix-verified against a standalone dnsmasq 2.90 with an unreachable upstream (type-65 REFUSED → NODATA).
- **First-boot hardening for the captive-portal window** (#483, #488). Shipped alongside #489 from the same investigation; useful defense-in-depth even though the DNS forward above (not readiness) turned out to be the actual cause. (1) `first-boot.sh` now waits for the setup server to be listening on port 8080 (bash `/dev/tcp` poll, 25s cap, best-effort) **before** painting the hotspot QR, so the user's join always lands after the server can answer a probe. (2) The image build now **ships pre-compiled bytecode** for `src/` (`compileall --invalidation-mode checked-hash`, compiled with the runtime venv interpreter, then chowned to `pi`) instead of deleting all `__pycache__` in the finalize stage, so a fresh flash no longer compiles the setup path cold on its first run; `checked-hash` keeps the image reproducible and correct across `update.sh`. A cross-model `/review` caught that the compile runs as root after the build's `chown` (would have shipped root-owned `.pyc` the `pi` runtime couldn't refresh after an OTA) — closed with a re-chown.

### Security
- **control_server now sends `X-Content-Type-Options: nosniff` + `X-Frame-Options: DENY` on every response** (#477). Added off the back of a whole-codebase SAST + DAST pass (report: `docs/security-sast-dast-2026-07.md`), which found the code otherwise clean — 0 dependency CVEs (pip-audit + npm audit), no SSRF (fixed hosts, quoted queries), no command injection (fixed-argv `sudo`/`systemctl`; the diagnostics `unit` arg is allowlist-gated), and CSPRNG (`secrets`) for the WiFi hotspot password + CSRF + confirm tokens — but the PWA served no HTTP hardening headers. `nosniff` is universal defense-in-depth; `X-Frame-Options: DENY` closes a clickjacking vector, since the PWA is unauthenticated on the LAN and a malicious same-LAN page could otherwise iframe it and trick the owner into a state-changing control (Reset WiFi / power off). Deliberately **not** applied to `setup_server` — the first-boot captive portal may render inside an OS captive-portal WebView, which `DENY` would break. A content-security-policy is deferred to the remote-exposure tripwire (where auth + HTTPS + CSP all become required together).

### For contributors
- **`setup_server` background threads are now joined between tests, killing a `test_wifi_retry_flow` CI flake** (#478, #479, #480). The WiFi connect/teardown handlers spawn daemon threads (`_delayed` SIGTERM timer, `_connect_and_teardown`, `_resolve_and_signal`) that write module globals + call test-monkeypatched functions. `reset_state()` (the #355 isolation helper) previously only polled the `WIFI_CONNECT_IN_FLIGHT` flag — which `_delayed`/`_resolve_and_signal` never set — so a prior test's thread could outlive its case and fire its (now next-test-monkeypatched) `os.kill`/retry into the next test's counters (intermittent `assert 4 == 3` / `assert 1 == 0` failures in CI while passing in isolation). It now tracks every spawned thread via a `_spawn_bg` registry and **joins** them (`start()` inside the lock to avoid a join-before-start `RuntimeError`; a keep-alive filter so a thread that outruns the join budget isn't forgotten), and `_delayed` uses a cancellable `_BG_CANCEL.wait(delay)` so `reset_state` cancels the timer without firing a real SIGTERM (conftest reverts `os.kill` before the reset fixture runs, so a fired signal would kill the runner). Regression tests assert on their own thread handle, not the shared registry count, so they can't re-introduce cross-test coupling. A post-merge cross-model `/review` of the first cut caught two latent races that the follow-up hardened.
- **Captive-portal probes now log Host + path + User-Agent in provisioning mode** (#483, #486). The setup server's access log recorded only the request line + status, which couldn't pin why an iOS phone's captive-portal sheet sometimes doesn't auto-open on join (observed in fresh-flash QA; tapping the browser manually always worked). A provisioning-only `CAPTIVE-PROBE` diagnostic now logs which host/path/UA each probe hit — paired with dnsmasq's already-enabled `log-queries`, a phone-join repro captures the full DNS→HTTP probe path so the root cause is pinnable. Diagnostic only; no change to the captive-portal response flow. Investigation (#483) confirmed the server side is correct (iOS probes get the non-"Success" 200 CNA bridge), so the miss is iOS-side probe timing/behavior — still open pending the repro logs.

## [v0.215.0] - 2026-06-25

### Changed
- **Settings → Advanced toggles now auto-save on tap (no Save button), matching the Weather toggle + Temperature pill** (#456). The Advanced section ("Allow NSFW quotes" / "Show diagnostics shortcut") was the last discrete-control holdout still rendering a Save button — a violation of DESIGN.md's locked "Save-button rule" (#337 A13: toggles/pills auto-save on change, only text/coords get an explicit Save). Each toggle now commits on change via the shared `autoSavePatch` path (AbortController + 504 `env_lock_timeout` handling) and **reverts to the last server-confirmed value** if the save fails — so the visual never claims a state the device didn't persist, even under rapid taps (`/review` cross-model HIGH: a `!newValue` revert could show the opposite of what's stored). Each toggle saves ONLY its own field as a JSON PATCH, so saving one can't flip the sibling (the JSON path writes only keys present in the body; missing-checkbox→false synthesis is form-path-only). The Save button drops to a no-JS fallback (`data-no-js-only`, CSS-hidden under `html.has-js`). A deeper pre-existing race shared by all three auto-save controls — an aborted request whose POST already reached the Pi commits after the UI moved on — was tracked in #457 and is now fixed (see the Fixed entry below; the fix turned out to be client-side serialization, not the server-side sequencing this note first guessed), and converging the Weather toggle + Temperature pill onto the hardened helper was #458 (folded into the same change).
- **`/diagnostics` "Not yet collected" tier now keys off a persistent marker, killing the post-reboot grey flicker** (#445, v0.215 follow-up to #432). The v0.214.4 grey tier asked "is the tmpfs `/run/litclock/last-rendered-ip` file present right now" — but that file is wiped at every boot, so a healthy clock flashed grey "Not yet collected" for ~5–10 s after each reboot until the NM dispatcher re-fired. The predicate now asks the honest question, "has this section ever been collected on this Pi," via a persistent JSON marker at `/var/lib/litclock/.last-collected-marker.json` (`{"network": "<iso>", "time-location": "<iso>"}`) that survives reboot. Writers: `scripts/litclock-mark-collected.sh` (the NM dispatcher records `network`, as root) and `src/collected_marker.py` (the IP-geo resolvers record `time-location`, as pi) — both share one file + sidecar lock + format, interlocked by `flock`. **Backward-compatible:** when the marker is absent or unreadable, `_compute_uncollected` falls back to the exact v0.214.4 behavior (tmpfs existence for `network`, env-only gate for `time-location`), so an existing Pi behaves unchanged until its first dispatcher / IP-geo write lands the persistent marker. No sudoers change needed — `/var/lib/litclock` is already `0755 pi pi`.

### Fixed
- **Settings auto-save controls can no longer persist state opposite the UI under rapid taps** (#457, with the #458 helper convergence). All three auto-save controls (Weather toggle, Temperature pill, Advanced toggles) used to `abort()` the in-flight `fetch` on a re-tap — but an aborted request whose POST already reached the Pi still completed its `env.sh` write, and writes serialize on the flock by acquisition order, not send order, so an older save could land *after* a newer one and persist the opposite of what the toggle showed (until the next reload). The fix is client-side **serialization + coalescing**: each control now keeps at most one save in flight and coalesces rapid taps to the latest desired value, firing a single follow-up save only after the in-flight one settles. Because two saves for one control never overlap, the flock can't reorder them — persisted state always converges to the value the user landed on (a burst of N taps costs ≤2 round-trips). On failure the helper reverts to the last server-confirmed value and alerts. As part of the same change (#458) the Weather toggle and Temperature pill moved off their hand-rolled `AbortController` handlers onto two shared helpers — `wireBooleanToggleAutoSave` (checkboxes) and the new `wireSegmentedAutoSave` (segmented pill) — so all four auto-save controls share two code paths instead of four near-identical bodies (~120 lines of duplication removed). Known residuals (all self-heal on reload): genuinely simultaneous edits from two *different* devices stay last-flock-wins, and a response lost *after* the server commits can't be reconciled client-side — full convergence would need server read-back, the reconcile alternative #457 weighed and set aside. Cross-model `/review` (Codex GPT-5.5 + testing/maintainability/adversarial Claude subagents) confirmed the fix and the no-reintroduced-F2 invariant; added `settings.weather-toggle.test.js` (the toggle had zero JS coverage) plus serialization/coalescing + revert-to-confirmed coverage for both helpers.
- **Oneshot service rows no longer read warning-ochre while the section pill reads OK** (#449, follow-up to #443). #443 stopped a `DIAG_ONESHOT_UNITS` member (`litclock.service`) mid-paint in `activating`/`deactivating` from tripping the services anomaly — the section header pill correctly reads **OK** in that ~2–5 s per-minute window. But the per-unit *row* state chip stayed warning-ochre (the `.diag-service__state--activating`/`--deactivating` CSS), so a user expanding the Services drawer at that instant saw an ochre `activating` chip on `litclock.service` under an OK section header — a small but real visual contradiction (exposed, not introduced, by #443: before it the ochre matched the anomaly). `_build_service_states` now emits a `state_modifier` per row that drives the chip *color* while the chip *text* stays the literal systemd state: a oneshot caught in `activating`/`deactivating` gets the neutral graphite `transient-ok` tone (matching the OK section tone), while non-oneshot `activating`/`deactivating` and any `failed` keep the warning-ochre tint they should. The oneshot carve-out is now a single shared predicate, `_is_oneshot_nonanomaly(unit, state)`, referenced by all three consumers (`_compute_anomalies` verdict, `_is_obviously_healthy` lazy-tail gate, and the row tone) so they can't drift.
- **Oneshot `litclock.service` lifecycle no longer false-positives the oxblood "Clock isn't running" banner** (#443). `litclock.service` is `Type=oneshot` and cycles `inactive → activating → active → inactive` in 2–5 s during every per-minute quote paint. A `/diagnostics` poll landing in the `activating`/`deactivating` window (~5–8 % of page loads by sample-rate math) tripped the services anomaly, which escalates the banner to its highest-severity error tier — "Clock isn't running." — even though the clock was *literally rendering the next quote at that second*. A genuinely wedged paint stuck in `activating` still surfaces — via the `last-quote` anomaly once `picked_at` ages past 90 s (covered by a regression test). Pre-existing since #416; surfaced by the v0.214.4 grey-tier work making `/diagnostics` a maintained support surface (caught during authorclock hardware QA 2026-06-13).

### For contributors

The PWA buttons appeared and accepted the user's confirm tap, but the backend `sudo systemctl start --no-block` then failed with `Unit not found`, surfacing as a generic error in the UI. Anyone flashing the v0.211.0 image hit this; `scripts/install.sh` (DIY install) + `scripts/update.sh` (in-place auto-update) paths were unaffected because both already copied the units correctly.

Caught in M8 hardware QA on test Pi 2026-05-13 — exactly the install-path bug class the fresh-image flash test was designed to catch.

Plus a copy nit caught in the same QA pass.

### Fixed
- `pi-gen/stage3/03-install-services/00-run.sh` — add `cp` for `litclock-wifi-reset.service` and `litclock-prepare-for-gift.service`. Both units are `Type=oneshot` with no `[Install]` section (fired on-demand via sudoers-allowed `systemctl start --no-block`), so no `systemctl enable` is needed at install time. Closes the fresh-image Reset WiFi + Prepare-for-Gifting outage.
- `src/control_server/templates/system.html.j2` — Reset WiFi confirm modal body: "Your **WiFi only** will be cleared" → "**Only your WiFi** will be cleared". Reads more naturally.

### Known limitations (carried forward from v0.211.0)

### Control PWA — M6 PWA shell + manifest + service worker (#245, #259)

The PWA now feels like an app on the home screen. Pin it once, get a real splash, real fonts, an offline cache (Android), and a friendly first-run "Add to Home Screen" nudge that knows when to keep quiet.

**What changed for the user:**
- iPhone cold launch shows a tinted splash within 250ms instead of a 2-second blank panel (closes #259) — both light and dark variants matched per `prefers-color-scheme`, no more cream-flash before the dark shell paints. 17 device sizes covered, every iPhone + iPad shipped in the last 5 years (iPhone SE 1st gen through iPhone 16/17 Pro Max + iPhone Air + iPad Pro 12.9").
- Self-hosted variable Fraunces / Instrument Sans / Geist Mono fonts (DESIGN.md "Typography" lock) — the literary serif/sans split lands by first paint with `font-display: swap`, system fallback for the ≤100ms before the woff2 arrives.
- First-run "Add to Home Screen" hint (phone-only, viewport-gated <600px). Bottom-sheet card slides up 600ms after page load with the iOS-native share-icon glyph (the actual icon users tap in Safari), Fraunces title + Instrument body + italic micro-instruction. Dismiss persists in localStorage; auto-dismisses without persisting if you navigate within the PWA.
- Android: full service worker. Pi rebooting? The shell still paints from cache. M5 update flow's "Restarting…" reload now lands on fresh code without a manual force-quit.

**Hardware-found platform limit (2026-04-30, post-merge Android QA):** the SW machinery is **inert on the v1 plain-HTTP origin** for both platforms, not just iOS. Per W3C spec, only `localhost` / `127.0.0.0/8` / `[::1]` are "potentially trustworthy" in plain HTTP. `192.168.x.x` is not, so both iOS Safari AND Android Chrome report `isSecureContext = false` and refuse to register the SW. The original M6 hardware probe only tested iPhone; Android was assumed from spec docs and turned out to be wrong. Net effect: cached-shell-during-reboot, offline rendering, and Android PWA-install all DON'T work today. The wiring ships intact (sw.js Flask route, manifest endpoint, sw-register.js feature-detect, 11 SW-specific tests) because (a) the visible M6 wins — cold-launch splash, fonts, AtHS hint, Dynamic Type — all work without SW, and (b) the wiring activates unmodified when v2 cloud-relay TLS lands. Full discussion in `docs/control-pwa-platform-limits.md`.

- New `routes/sw.py` blueprint serves `/sw.js` (templated `CACHE_NAME = 'litclock-{version}'` from `version.get_version()` so each release lands a fresh cache without manual bumping) + `/manifest.webmanifest` (with `Content-Type: application/manifest+json` + `Cache-Control: max-age=86400`).

- `tools/control-pwa/fetch_fonts.py` pins Fontsource versions + SHA256-verifies on each `--check` run; CI catches drift.
- `tools/control-pwa/generate_pwa_icons.py --splash` extends the icon generator to emit the 34-PNG splash matrix (logo at 50% of canvas shorter side per Apple HIG, light + dark canvases).
- `--fs-caption` ceiling raised 14 → 18px so iOS Dynamic Type can grow tab labels visibly past iOS Larger 1 (closes #258); M5's tactical local override on `.tabbar a` removed.
- NOTICE.md updated with SIL OFL entries for Fraunces, Instrument Sans, Geist Mono.

### Control PWA — M3 Settings tab (#245)

Edit your LitClock from a phone-pinnable web app, no SSH needed. Open the Settings tab on `litclock.local:8443`, change weather city, units, gift-mode message, or NSFW filter — taps land on the e-ink display within ~3 seconds, not 60. Five sections (Weather / Units / Gift Mode / Location / Advanced), no-JS forms work in any browser, JS-enabled flows get a live geocode preview as you type a city or zip.

**Under the hood:**
- `setup_server.save_settings` and the new control_server now share one writer (`config.atomic_update`), one validator, one lockfile. The first-boot flow and the post-boot Settings tab can no longer disagree about how `env.sh` gets written.
- Free-form values (gift-mode message, resolved location name) are wrapped via `shlex.quote()` at write time and content-allowlisted (no backtick, no `$`, 280-char cap on gift mode).
- Server-side geocoding via Nominatim with IP-country biasing — type "78701" and you get Austin, TX, not a foreign postal code. Timezone updates automatically when coords change.
- CSRF protection via reflexive Origin/Referer match against `request.host` (works for `litclock.local`, raw IP, or whatever name your LAN's mDNS resolves first) plus a multi-use synchronizer token (action=settings, TTL 30 min).
- New env keys: `WEATHER_ENABLED`, `WEATHER_LOCATION_NAME`, `GIFT_MODE_ENABLED`, `GIFT_MODE_MESSAGE`. The first-boot flow keeps writing the existing keys; new keys default to safe values until edited.
- `litclock.service` is `Type=oneshot` + timer-fired, so saves don't restart anything — they fire `systemctl start --no-block litclock.service` for one immediate tick. The minute-tick contract stays intact.

**Gift-mode runtime semantics** (rendering the message in place of a quote on the e-ink) is deferred to a follow-up issue. M3 ships the editing surface only.

### Heads-up — weekly auto-update behavior (#209)

LitClock now updates itself automatically **once a week on Sunday at 03:00 local time** (with up to 7 days of randomization to avoid everyone hitting GitHub at once). The appliance posture is deliberate: non-technical users should never have to think about updates, and power users can opt out in one command.

**What the weekly update does:**
- Pulls the latest blessed GitHub Release (`litclock-*` tag), not `origin/master`. A release is only cut once a SHA has been hardware-verified.
- Rebuilds the Python venv when `requirements.txt` changes.
- Runs a post-rebuild **smoke test** (`python -m src.literary_clock --dry-run`). If the render fails, the update auto-reverts to the previous SHA and the clock keeps ticking on the old code.
- Records a last-known-good SHA to `/var/lib/litclock/lkg-sha` after the clock has run 10 minutes on the new code (observability today; an opt-in auto-revert on repeated boot failures is a future issue).
- Merges new env-var names from `env.sh.sample` into your `env.sh` without overwriting your values.

**What does NOT update automatically:**
- OS packages (`apt upgrade`), Pi firmware — those need A/B partitioning we don't have yet.

**How to opt out:**
```bash
sudo systemctl disable --now litclock-update.timer
```

**How to inspect:**
```bash
systemctl status litclock-update.timer
journalctl -u litclock-update.service --since "1 week ago"
```

See the [Updating](README.md#updating) section of the README for the full story. Refs #209.

#### Auto-update — Added
- `scripts/lib/github_api.sh` — shared GitHub REST API helpers (auth, timeout, graceful-offline). Used by `update.sh` and `download_images.sh`.
- `resolve_target_sha()` in `scripts/update.sh` — resolves the latest non-prerelease Release tag to an authoritative SHA via `git rev-list`. Reads `tag_name` (not the unreliable `target_commitish`).
- Phase 4.5 smoke-test block in `update.sh` — runs `python -m src.literary_clock --dry-run` between venv rebuild and systemd unit sync, auto-reverts on failure, writes `/var/lib/litclock/update-failed` (cleared on next successful update).
- `--dry-run` flag on `src/literary_clock.py` — renders a quote to an in-memory PIL image without touching GPIO/SPI. Used by the update smoke test.
- Subtle "!" glyph in the top-right corner of the clock face when the last auto-update failed. Self-clearing.
- `systemd/litclock-update.service` + `.timer` — weekly oneshot with `TimeoutStartSec=120`, `OnCalendar=Sun 03:00`, `Persistent=true`, `RandomizedDelaySec=7d`.
- `systemd/litclock-lkg.service` + `scripts/litclock-lkg-record.sh` — LKG heartbeat writer. Records HEAD SHA after 10 minutes of clean clock runtime. Writer-only in v1; auto-revert consumer deferred (see `TODOS.md`).
